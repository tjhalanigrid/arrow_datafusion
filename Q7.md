# Q7: DataFusion can read CSV, JSON, and Parquet files. How does it convert these external formats into Arrow RecordBatches without materializing the entire dataset?

DataFusion reads external file formats incrementally, producing `RecordBatch` values of bounded size (default: 8192 rows) and streaming them through the execution pipeline. The entire dataset is never fully loaded into memory. This works because the execution engine is built around a *pull-based streaming model* where operators request one batch at a time.

The paper states:
> "Whenever possible, all operators produce output incrementally as Arrow Arrays grouped into RecordBatches, with a default size of 8192 rows. For pipeline-breaking operations such as a full sort, final aggregation, or a hash join, the operators buffer tuples, spilling to disk if necessary."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 5.5.1

---

## The streaming execution model

Every DataFusion operator implements the Rust `Stream` trait, producing `RecordBatch` values on demand:

Source: [`datafusion/execution/src/stream.rs#L26`](https://github.com/apache/datafusion/blob/main/datafusion/execution/src/stream.rs#L26)

```rust
pub trait RecordBatchStream: Stream<Item = Result<RecordBatch>> {
    fn schema(&self) -> SchemaRef;
}

pub type SendableRecordBatchStream = Pin<Box<dyn RecordBatchStream + Send>>;
```

An operator only produces a batch when the consumer calls `poll_next()` on the stream. This is the Volcano pull model: data flows upward through the plan tree, one batch at a time.

From the SIGMOD paper (Figure 3):

```rust
impl Stream for MyOperator {
    // Pull next input (may yield at await)
    while let Some(batch) = stream.next().await {
        // Calculate, check if output is ready
        if Some(output) = self.process(&batch)? {
            // "Return" RecordBatch to output
            tx.send(batch).await
        }
    }
}
```

## How format conversion works: FileStream

The common streaming infrastructure for all file formats is `FileStream`:

Source: [`datafusion/datasource/src/file_stream.rs#L46`](https://github.com/apache/datafusion/blob/main/datafusion/datasource/src/file_stream.rs#L46)

```rust
/// A stream that reads from a single or multiple files and converts them
/// to RecordBatches, one file at a time, batch by batch.
pub struct FileStream {    // L46
    /// An iterator over input files (not all loaded at once)
    file_iter: VecDeque<PartitionedFile>,
    /// The currently open file reader (None between files)
    state: FileStreamState<F>,
    /// The configured batch size
    batch_size: usize,
    /// Optional limit on total rows to read
    remain: Option<usize>,
    // ...
}
```

`FileStream` implements `Stream<Item = Result<RecordBatch>>`. It opens one file at a time, reads batches from that file until exhausted, then opens the next file. At no point are all files open simultaneously, and at no point is any entire file loaded into memory.

The `remain` field implements LIMIT pushdown: once enough rows have been produced, the stream stops opening further files. This is part of DataFusion's pushdown strategy described in the paper (Section 6.8).

## Format-specific batch readers

### CSV

Source: [`datafusion/datasource-csv/src/source.rs#L191`](https://github.com/apache/datafusion/blob/main/datafusion/datasource-csv/src/source.rs#L191)

```rust
let builder = csv::ReaderBuilder::new(Arc::clone(self.table_schema.file_schema()))  // L191
    .with_batch_size(
    .with_projection(Some(projection))
    .with_bounds(config.start, config.end);
let reader = builder.build(file)?;
```

The Arrow CSV reader is constructed with `batch_size` -- it reads exactly that many rows per batch from the byte stream. Rows are decoded and converted directly to Arrow columns without building an intermediate row-oriented representation.

The paper notes (Section 8.1) that DataFusion benefits from *"the optimized CSV parser included in the Rust implementation of Apache Arrow"*, contributing to its H2O-G benchmark performance.

### Parquet (with late materialization and predicate pushdown)

Parquet is the most sophisticated case because the format has rich metadata that enables DataFusion to skip large amounts of work.

Source: [`datafusion/datasource-parquet/src/opener.rs#L1035`](https://github.com/apache/datafusion/blob/main/datafusion/datasource-parquet/src/opener.rs#L1035)

```rust
// Use an unfold-based stream to lazily decode row groups
let stream = futures::stream::unfold(state, |mut state| async move {  // L1035
    match state.next_row_group().await {
        Ok(Some(batch)) => Some((Ok(batch), state)),
        Ok(None) => None,
        Err(e) => Some((Err(e), state)),
    }
});
```

The `unfold` pattern creates a lazy stream: the next row group is decoded only when the consumer calls `poll_next()`. Row groups not yet consumed are not decoded.

The paper describes DataFusion's Parquet late materialization pipeline (Section 6.8):

1. **Row Group pruning**: Skip entire row groups where `A_max <= 35` or `B_min > "F"` using embedded metadata
2. **Column B decode + filter**: Decode only column B first, evaluate `B = "F"`, capture matching row indices as a `RowSelection`
3. **Column A page-level decode**: Use the Page Index to decode only pages containing matching rows from column A, evaluate `A > 35`
4. **Remaining columns**: Decode only the surviving rows for any other projected columns

This technique (late materialization) was studied by Abadi et al. (2007, cited as [1] in the SIGMOD paper) and is especially effective when predicate columns appear early in sort order.

Source: [`datafusion/datasource-parquet/src/opener.rs`](https://github.com/apache/datafusion/blob/main/datafusion/datasource-parquet/src/opener.rs)

```rust
// Apply pushed-down predicates to the Parquet reader
let mut row_group_reader = file_reader.get_row_group(rg_idx)?;

// Stage 1: Evaluate row-group-level statistics against pushed predicates
if let Some(ref pruning_stats) = self.pruning_statistics {
    if pruning_stats.should_prune(rg_metadata)? {
        continue; // skip entire row group
    }
}

// Stage 2: RowSelection from column-level filtering (bloom filters, page index)
if let Some(selection) = self.build_row_selection(row_group_reader)? {
    // Only decode selected rows for remaining columns
}
```

### JSON

JSON readers follow the same `FileStream` pattern with Arrow's streaming JSON decoder, which produces `RecordBatch` values incrementally and handles schema inference from the first batch.

## Batch size configuration

`DataSourceExec` threads the configured `batch_size` through to each format reader:

Source: [`datafusion/datasource/src/file_scan_config.rs#L476`](https://github.com/apache/datafusion/blob/main/datafusion/datasource/src/file_scan_config.rs#L476)

```rust
pub fn with_batch_size(mut self, batch_size: Option<usize>) -> Self {
    self.batch_size = batch_size;
    self
}
```

The default `batch_size` is 8192 rows (configurable via `SessionConfig::with_batch_size`). This controls the granularity of the streaming: no batch is larger than this. For wide tables with many columns, even 8192 rows might consume substantial memory, so users can tune this value.

Source: [`datafusion/datasource/src/source.rs#L264`](https://github.com/apache/datafusion/blob/main/datafusion/datasource/src/source.rs#L264)

```rust
pub struct DataSourceExec {   // L264
    /// The source of the data -- e.g., FileScanConfig or MemorySourceConfig
    data_source: Arc<dyn DataSource>,
    /// Cached plan properties such as sort order
    cache: Arc<PlanProperties>,
}
```

## Schema inference without materialization

For CSV and JSON, DataFusion can infer schema by reading only the first batch:

```rust
// CSV schema inference: read a small sample, infer types
let format = CsvFormat::default().with_schema_infer_max_rec(Some(100));
let schema = format.infer_schema(&store, &objects).await?;
```

The full dataset is not read during schema inference -- only enough rows to determine column types.

## Predicate and projection pushdown at the scan level

Even before the first batch is read, `DataSourceExec` applies pushdown:

Source: from `TableProvider::scan()` contract:

```rust
async fn scan(
    &self,
    state: &dyn Session,
    projection: Option<&Vec<usize>>,  // only these columns needed
    filters: &[Expr],                 // pushed-down predicates
    limit: Option<usize>,             // stop after this many rows
) -> Result<Arc<dyn ExecutionPlan>>;
```

- **Projection**: Only selected columns are decoded. For Parquet and CSV, unused columns are skipped entirely.
- **Filters**: Passed to the file reader to enable row group pruning, bloom filters, and page-level filtering.
- **Limit**: `FileStream.remain` stops reading after the limit is reached.

This is why DataFusion performs well on ClickBench queries with highly selective predicates (Section 8.1 of the paper): *"DataFusion performs better on queries with highly selective predicates such as Q2, Q8, and Q20 likely due to its ability to push predicates into the parquet scan to skip entire row groups."*

---

## Memory management during streaming

The paper describes DataFusion's memory pool (Section 5.5.4). For pipeline-breaking operators that must buffer data (hash joins, sorts, aggregates), DataFusion:

1. Tracks memory usage via `MemoryPool.grow()` / `shrink()` calls
2. Spills accumulated data to temporary disk files when the memory budget is exhausted
3. Merges spilled files during the final output phase

Scan operators (CSV, Parquet, JSON) are *not* pipeline-breaking: they produce one batch at a time and do not accumulate state. Each batch is processed and discarded by downstream operators before the next is read.

---

## Summary

DataFusion converts external formats to Arrow `RecordBatch` streams without full materialization through: a pull-based Volcano streaming model (one batch at a time), the `FileStream` abstraction that opens files lazily, format-specific readers (`batch_size`-bounded decoders), and aggressive pushdown of predicates/projections/limits into the scan layer. Parquet additionally benefits from row group pruning, bloom filters, and late materialization to minimize decoded data. Together, these techniques allow DataFusion to process datasets far larger than available memory.
