# Q9: Join ordering (deciding which tables to join first in a multi-table query) is important for performance. Why is it so hard to get right, and how does DataFusion currently approach it?

Join ordering is NP-hard in the general case, sensitive to cardinality estimation errors, and computationally expensive even for moderate numbers of tables. DataFusion currently uses pragmatic heuristics -- pairwise size-based swapping and cross-join elimination -- rather than full dynamic programming enumeration. This is a known limitation that the DataFusion project is actively working to improve.

The paper is explicit about this limitation:
> "There are also several queries where DataFusion is well over 2x slower, such as Q11, Q17, Q18, and Q21. Much of this difference is due to a suboptimal join order, and when we manually force a better join order, the performance of the two systems becomes similar."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 8.1

---

## Why join ordering is hard

### 1) The search space is factorial

For `n` tables in a query, the number of possible left-deep join orders is `n!`. For `n=5`, that's 120 orderings; for `n=10`, it's 3,628,800. The Selinger (System-R) dynamic programming algorithm reduces this to `O(2^n)` by building optimal subplans bottom-up, but even this is exponential and impractical for queries with 15+ tables.

Additionally, if we allow *bushy* trees (not just left-deep or right-deep), the search space is much larger still.

### 2) Join ordering quality depends entirely on cardinality estimates

The optimal join order is the one that minimizes the total amount of data flowing through intermediate joins. This requires knowing:
- How many rows each table produces after filtering
- How much a join reduces or amplifies cardinality (the join fanout)

Both of these depend on cardinality estimation -- which, as discussed in Q8, is fundamentally difficult. Leis et al. (2015) showed that real optimizers frequently misestimate join cardinalities by orders of magnitude, making "optimal" plans computed from bad estimates actively worse than simple heuristics in some cases.

### 3) Cross-join patterns arise from SQL expansions

When a SQL query has multiple tables in the `FROM` clause without explicit `JOIN` syntax, the parser generates cross joins initially:

```sql
SELECT * FROM orders o, customers c, products p
WHERE o.customer_id = c.id AND o.product_id = p.id
```

This becomes a `CrossJoin(CrossJoin(orders, customers), products)` at the logical plan level. The `EliminateCrossJoin` optimizer rule must detect the filter predicates and convert these to `InnerJoin` with the appropriate equijoin conditions before any ordering decision can be made.

### Step 2: should_swap_join_order (JoinSelection physical optimizer)

Source: [`datafusion/physical-optimizer/src/join_selection.rs#L58`](https://github.com/apache/datafusion/blob/main/datafusion/physical-optimizer/src/join_selection.rs#L58)

```rust
/// Decides if left and right sides of a join should be swapped.
/// Places the smaller side on the build (left) side of a hash join.
/// Uses byte size first, falls back to row count, then gives up.
pub(crate) fn should_swap_join_order(
    left: &dyn ExecutionPlan,
    right: &dyn ExecutionPlan,
    config: &ConfigOptions,
) -> Result<bool> {
    if !config.optimizer.join_reordering {
        return Ok(false);  // feature can be disabled entirely
    }
    let left_stats = left.partition_statistics(None)?;
    let right_stats = right.partition_statistics(None)?;
    match (
        left_stats.total_byte_size.get_value(),
        right_stats.total_byte_size.get_value(),
    ) {
        (Some(l), Some(r)) => Ok(l > r),
        _ => match (
            left_stats.num_rows.get_value(),
            right_stats.num_rows.get_value(),
        ) {
            (Some(l), Some(r)) => Ok(l > r),
            _ => Ok(false),
        },
    }
}
```

This is a **pairwise** heuristic operating on one join at a time: it puts the smaller side on the build side of a hash join to minimize hash table memory. It does not globally enumerate orderings for multi-table queries.

The optimizer config that controls this:

Source: [`datafusion/common/src/config.rs`](https://github.com/apache/datafusion/blob/main/datafusion/common/src/config.rs)

```rust
/// When set to true, the physical optimizer will attempt to reorder joins
/// to put the smaller side as the build side of a hash join.

```

### Step 3: ExtractEquijoinPredicate (logical rule 9)

This rule extracts equijoin predicates from `WHERE` clause filters and moves them into the `ON` clause of the join, enabling the hash and sort-merge join implementations to recognize equijoin keys.

## Why DataFusion doesn't use full Selinger DP enumeration

The Selinger (System-R, 1979) dynamic programming approach enumerates all subsets of tables and finds the optimal join order with O(2^n) time and space. DataFusion's pragmatic decision to use heuristics instead reflects several practical considerations:

1. **Cardinality estimates are unreliable**: A perfect enumeration with imperfect statistics can produce a worse plan than a simple heuristic with the right fallback. The paper acknowledges this indirectly by noting that manually forcing a better join order (not the optimizer's choice) closes the performance gap with DuckDB.

2. **Optimization time vs. query time**: For small queries (3-4 tables), any ordering strategy is fast. For large queries (10+ tables), O(2^n) DP may take longer than the query itself for interactive workloads.

3. **Extensibility over completeness**: DataFusion prioritizes being a correct, extensible engine over being an optimizer research platform. Advanced join ordering can be added as an `OptimizerRule` without changing the core.

The paper notes this as an open area (Section 8.1, footnote 4): the join ordering issue is tracked as a known limitation in the GitHub issue tracker.

## What a full implementation would require

For comparison, the Selinger DP approach:

```
Best[{R}] = scan cost for R
For size k = 2 to n:
  For each subset S of size k:
    For each table T in S:
      plan = Best[S - {T}] ⋈ T
      cost = cost(plan) + C_join(|Best[S-{T}]|, |T|)
      if cost < Best[S]: Best[S] = plan
Return Best[{all tables}]
```

This requires:
- Accurate cardinality estimates for all intermediate joins
- A cost model that accounts for I/O, memory, and CPU
- Consideration of interesting orders (sorts that benefit later operators)
- Pruning strategies to handle bushy tree exploration

The last point (interesting orders) was part of the original Selinger paper but is absent from DataFusion's current optimizer. DataFusion does track sort orders through `EquivalenceProperties` on `ExecutionPlan`, but does not yet feed these back into join ordering decisions.

---

## Summary

Join ordering is hard because: the search space is factorial (reduced to `2^n` with DP), quality depends on cardinality estimates that are often wrong, and the benefits must outweigh the optimization cost. DataFusion currently uses pairwise heuristics (`should_swap_join_order`) and cross-join elimination (`EliminateCrossJoin`) rather than full DP enumeration. The paper explicitly acknowledges this as a performance gap -- several TPC-H queries are 2x slower than DuckDB specifically due to suboptimal join ordering, with improvements underway in the community.
