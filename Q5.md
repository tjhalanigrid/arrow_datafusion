# Q5: DataFusion's optimizer applies logical transformations in a specific order (e.g., predicate pushdown before column pruning). Why is order important, and what would happen if transformations were applied in arbitrary order?

The order in which optimizer rules execute determines which optimization opportunities are visible to each rule. DataFusion explicitly documents that rule ordering matters and encodes dependency comments directly in the code.

The paper describes DataFusion's rewrite framework as:
> "DataFusion includes an extensible plan rewriting framework, implemented as a series of LogicalPlan and ExecutionPlan rewrites. These passes handle details such as automatically coercing types to match available operator and function signatures, and introducing necessary sort and redistribution operations."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 5.4.4

---

## The exact logical optimizer rule ordering (24 rules)

Source: [`datafusion/optimizer/src/optimizer.rs#L268`](https://github.com/apache/datafusion/blob/main/datafusion/optimizer/src/optimizer.rs#L268)

```rust
impl Optimizer {
    pub fn new() -> Self {
        // NOTE: The order of rules in this list is important, as it determines
        // the order in which they are applied. Adding a new rule here is expensive
        // as it will be applied to all queries and will likely increase optimization time.
        let rules: Vec<Arc<dyn OptimizerRule + Sync + Send>> = vec![
            Arc::new(RewriteSetComparison::new()),            //  1 - simplify IN/NOT IN
            Arc::new(OptimizeUnions::new()),                   //  2 - combine adjacent UNIONs
            Arc::new(SimplifyExpressions::new()),              //  3 - constant folding, canonicalize
            Arc::new(ReplaceDistinctWithAggregate::new()),     //  4 - DISTINCT => GROUP BY
            Arc::new(EliminateJoin::new()),                    //  5 - always-false join => Empty
            Arc::new(DecorrelatePredicateSubquery::new()),     //  6 - flatten correlated subqueries
            Arc::new(ScalarSubqueryToJoin::new()),             //  7 - scalar subquery => join
            Arc::new(DecorrelateLateralJoin::new()),           //  8 - lateral join decorrelation
            Arc::new(ExtractEquijoinPredicate::new()),         //  9 - extract a=b from ON clause
            Arc::new(EliminateDuplicatedExpr::new()),          // 10 - remove duplicate GROUP BY keys
            Arc::new(EliminateFilter::new()),                  // 11 - always-true/false filters
            Arc::new(EliminateCrossJoin::new()),               // 12 - cross join => inner join
            Arc::new(EliminateLimit::new()),                   // 13 - LIMIT 0 => Empty
            Arc::new(PropagateEmptyRelation::new()),           // 14 - propagate empty inputs upward
            Arc::new(FilterNullJoinKeys::default()),           // 15 - add IS NOT NULL for join keys
            Arc::new(EliminateOuterJoin::new()),               // 16 - outer => inner when safe
            Arc::new(PushDownLimit::new()),                    // 17 - push LIMIT toward scans
            Arc::new(PushDownFilter::new()),                   // 18 - predicate pushdown
            Arc::new(SingleDistinctToGroupBy::new()),          // 19 - single DISTINCT => GROUP BY
            Arc::new(EliminateGroupByConstant::new()),         // 20 - remove constant GROUP BY keys
            Arc::new(CommonSubexprEliminate::new()),           // 21 - CSE
            Arc::new(ExtractLeafExpressions::new()),           // 22 - prepare projections
            Arc::new(PushDownLeafProjections::new()),          // 23 - push projections down
            Arc::new(OptimizeProjections::new()),              // 24 - final column pruning
        ];
        Self::with_rules(rules)
    }
}
```

## Why order matters: documented dependencies

### Rule 17 before Rule 18 -- `PushDownLimit` before `PushDownFilter`

Source: [`datafusion/optimizer/src/optimizer.rs#L297`](https://github.com/apache/datafusion/blob/main/datafusion/optimizer/src/optimizer.rs#L297)

```rust
// Filters can't be pushed down past Limits, we should do PushDownLimit before PushDownFilter
Arc::new(PushDownLimit::new()),   // 17
Arc::new(PushDownFilter::new()),  // 18
```

A `LIMIT 10` restricts how many rows are returned. If a filter were pushed *below* a limit, the semantics would change: the filter might eliminate rows that should have counted toward the limit.

**Example**: `SELECT * FROM t LIMIT 10 WHERE status = 'A'` -- the limit applies to the pre-filter result set. Pushing the filter below the limit first could produce a different 10 rows.

### Rules 6-8 before Rules 9, 12 -- Subquery decorrelation before join rewrites

`DecorrelatePredicateSubquery` (6), `ScalarSubqueryToJoin` (7), and `DecorrelateLateralJoin` (8) convert correlated subqueries into explicit join nodes. Only *after* these rules create those join nodes can:
- `ExtractEquijoinPredicate` (9) extract equality predicates from the new join's `ON` clause
- `EliminateCrossJoin` (12) detect and convert cross joins among the new joins

If these were reversed, there would be no join nodes for rules 9 and 12 to operate on.

### Rule 3 early -- `SimplifyExpressions` before everything else

Expression simplification (constant folding, canonicalization) runs at position 3 so that all subsequent rules operate on canonical, simplified forms. Benefits:
- `PushDownFilter` can identify simpler predicates to push
- `EliminateFilter` can detect constant `true`/`false` after simplification
- `CommonSubexprEliminate` can identify CSE candidates more reliably

**Example**: `1 + 1 > 2` simplifies to `false` before `EliminateFilter` eliminates the node entirely.

### Rules 22-24 last -- Projection preparation and column pruning run last

```rust
Arc::new(ExtractLeafExpressions::new()),   // 22
Arc::new(PushDownLeafProjections::new()),  // 23
Arc::new(OptimizeProjections::new()),      // 24
```

The code comments explain why:

```rust
// The previous optimizations added expressions and projections,
// that might benefit from the following rules
Arc::new(CommonSubexprEliminate::new()),
Arc::new(ExtractLeafExpressions::new()),
Arc::new(PushDownLeafProjections::new()),
Arc::new(OptimizeProjections::new()),
```

Earlier rules (subquery decorrelation, filter pushdown, CSE) add new projection and expression nodes. Running `OptimizeProjections` before those rules would miss the final set of needed columns and either prune too aggressively (breaking the plan) or not enough (leaving unused columns in the pipeline). Running it last ensures it sees the final, stable column requirements.

## Analyzer phase ordering (runs before all optimizer rules)

The analyzer phase applies semantic analysis rules. These also have ordering requirements:

Source: [`datafusion/optimizer/src/analyzer/mod.rs#L85`](https://github.com/apache/datafusion/blob/main/datafusion/optimizer/src/analyzer/mod.rs#L85) (`impl Analyzer` at L85, `pub fn new()` at L87, `let rules:` at L88)

```rust
impl Analyzer {                                           // L85
    pub fn new() -> Self {                               // L87
        let rules: Vec<Arc<dyn AnalyzerRule + Send + Sync>> = vec![  // L88
            Arc::new(InlineTableScan::new()),
            Arc::new(TypeCoercion::new()),
            // ...
        ];
        Self::with_rules(rules)
    }
}
```

And the function rewrite dependency documented at L137 in the source:

```rust
// Note this is run before all other rules since it rewrites based on the
// argument types (List or Scalar), and TypeCoercion may cast the argument
// types from Scalar to List.
Arc::new(ApplyFunctionRewrites::new(...)),
Arc::new(TypeCoercion::new()),
```

`ApplyFunctionRewrites` must run before `TypeCoercion` because type coercion may convert scalar arguments to lists, which would invalidate the function signature checks that `ApplyFunctionRewrites` relies on.

## Physical optimizer ordering (21 rules, also ordered)

Source: [`datafusion/physical-optimizer/src/optimizer.rs#L93`](https://github.com/apache/datafusion/blob/main/datafusion/physical-optimizer/src/optimizer.rs#L93)

```rust
let rules: Vec<Arc<dyn PhysicalOptimizerRule + Send + Sync>> = vec![
    Arc::new(AggregateStatistics::new()),     //  1
    Arc::new(JoinSelection::new()),           //  2 - pick join algorithm
    Arc::new(LimitPushdown::new()),           //  3 - push limits
    Arc::new(ProjectionPushdown::new()),      //  4
    Arc::new(FilterPushdown::new()),          //  5 - static filter pushdown (early)
    Arc::new(EnforceDistribution::new()),     //  6 - add Repartition nodes
    Arc::new(CombinePartialFinalAggregate::new()), // 7
    Arc::new(EnforceSorting::new()),          //  8 - add Sort nodes
    Arc::new(TopKRepartition::new()),         //  9
    Arc::new(CombinePartialFinalAggregate::new()), // 10 (second pass)
    Arc::new(OptimizeAggregateOrder::new()),  // 11
    Arc::new(PipelineFixer::new()),           // 12
    Arc::new(FilterPushdown::new()),          // 13 - dynamic filter pushdown (late)
    Arc::new(OutputRequirements::new()),      // 14
    Arc::new(SanityCheckPlan::new()),         // 15 - final validation, runs last
];
```

Key documented physical ordering constraints:

1. **`JoinSelection` (2) before `EnforceDistribution` (6)**: Join algorithm selection determines what data distribution requirements exist. Distribution enforcement must know if we're using `PartitionMode::Partitioned` (needs repartition) or `CollectLeft` (doesn't).

2. **`EnforceDistribution` (6) before `EnforceSorting` (8)**: Adding repartition nodes (`RepartitionExec`) can break existing sort orders. Sort enforcement must run after repartitioning so it can see which sorts still need to be added.

3. **`LimitPushdown` (3) before `TopKRepartition` (9)**: `TopKRepartition` needs `SortExec` nodes to already have `fetch` (TopK limit) set before it can add per-partition TopK operators.

4. **`FilterPushdown` runs twice**: First at position 5 for static filters (safe to push without concern for dynamic references), and again at position 13 for dynamic (runtime) filters after `EnforceDistribution` has finalized the plan topology.

5. **`SanityCheckPlan` runs last**: Validates physical correctness of the final plan (no unresolved requirements, valid partitioning). Only meaningful after all other rules complete.

## What happens with arbitrary ordering

| Ordering violation | Consequence |
|---|---|
| Filter pushed past Limit | Changed query semantics -- wrong results |
| Column pruning before predicate pushdown | Prunes columns that predicates need -- plan error |
| Sort enforcement before distribution | Sorts added that distribution would eliminate; or missing sorts after repartition |
| Subquery decorrelation after join extraction | No join nodes to extract predicates from |
| Type coercion before function rewrites | Wrong function signatures -- runtime errors |
| Simplification late | Later rules match non-canonical forms redundantly |
| SanityCheck not last | Validates an intermediate plan, misses downstream errors |

Rules are not independent; they form a dependency graph. The ordered list encodes a topological sort of that graph.

---

## Summary

Optimizer rule order is not arbitrary -- it encodes semantic dependencies (limit vs. filter), producer-consumer relationships (subquery decorrelation creates join nodes for downstream rules), and efficiency concerns (simplify early, prune late). DataFusion's explicit ordered rule lists ensure correctness, maximize optimization opportunities, and prevent rule conflicts. Both the logical optimizer (24 rules) and physical optimizer (21 rules) have documented ordering constraints that are enforced by their fixed execution sequence.
