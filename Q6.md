# Q6: DataFusion maintains both a LogicalPlan and an ExecutionPlan. Why are there two separate plan representations rather than a single unified plan?

DataFusion uses two separate plan representations because they serve fundamentally different purposes: `LogicalPlan` describes *what* a query computes (relational algebra), while `ExecutionPlan` describes *how* to compute it (concrete algorithms, parallelism, memory, sort order). Conflating these responsibilities into a single structure would compromise both optimization quality and architectural clarity.

The paper describes this layered architecture as central to DataFusion's design:
> "A Front End creates a tree of relational operators and expressions, called a LogicalPlan. Optimizers rewrite the LogicalPlan to a more optimal form. The LogicalPlan is lowered to an ExecutionPlan that includes characteristics of the intermediate results such as sort order, and specific algorithm selections. Additional optimizers rewrite the ExecutionPlan to match available physical resources and data layout. The ExecutionPlan is executed by Streams that incrementally produce results."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 5.1

## The two representations compared

### LogicalPlan: Abstract relational algebra

Source: [`datafusion/expr/src/logical_plan/plan.rs#L206`](https://github.com/apache/datafusion/blob/main/datafusion/expr/src/logical_plan/plan.rs#L206)

```rust
pub enum LogicalPlan {
    Projection(Projection),
    Filter(Filter),
    Window(Window),
    Aggregate(Aggregate),
    Sort(Sort),
    Join(Join),
    CrossJoin(CrossJoin),
    Repartition(Repartition),
    Union(Union),
    TableScan(TableScan),
    EmptyRelation(EmptyRelation),
    Subquery(Subquery),
    SubqueryAlias(SubqueryAlias),
    Limit(Limit),
    Statement(Statement),
    // ... ~15 variants total
}
```

`LogicalPlan` variants are abstract -- `Join` does not say whether it's a hash join, a sort-merge join, or a nested loop join. `TableScan` does not specify how many partitions or what batch size. A `Filter` node does not know whether it runs single-threaded or across 16 partitions.

Each `LogicalPlan` node also carries a `Schema` (output column types and names) but no information about physical properties like sort order or data distribution.

### ExecutionPlan: Concrete executable operator

Source: [`datafusion/physical-plan/src/execution_plan.rs#L96`](https://github.com/apache/datafusion/blob/main/datafusion/physical-plan/src/execution_plan.rs#L96)

```rust
pub trait ExecutionPlan: Debug + DisplayAs + Send + Sync {
    fn name(&self) -> &str;
    fn schema(&self) -> SchemaRef;
    fn properties(&self) -> &Arc<PlanProperties>;   // partition/sort/boundedness info
    fn children(&self) -> Vec<&Arc<dyn ExecutionPlan>>;
    fn execute(
        &self,
        partition: usize,
        context: Arc<TaskContext>,
    ) -> Result<SendableRecordBatchStream>;
    fn statistics(&self) -> Result<Statistics>;
    // ...
}
```

`ExecutionPlan` is a *trait* with dozens of concrete implementations, each with a specific algorithm, partitioning scheme, and execution behavior:

- `FilterExec` -- evaluates a predicate on each `RecordBatch`
- `HashJoinExec` (PartitionMode: Partitioned) -- builds sharded hash tables in parallel
- `SortMergeJoinExec` -- merges sorted streams
- `DataSourceExec` -- reads from a `TableProvider` with pushdown parameters
- `RepartitionExec` -- redistributes data across partitions (Volcano exchange operator)
- `SortExec` -- external sort with spill-to-disk
- `HashAggregateExec` -- two-phase partitioned hash aggregation

Each `ExecutionPlan` is associated with a `PlanProperties` struct:

```rust
pub struct PlanProperties {
    pub eq_properties: EquivalenceProperties,  // known sort orders, equivalences
    pub partitioning: Partitioning,             // how data is distributed
    pub output_ordering: Option<LexOrdering>,  // guaranteed output sort order
    pub boundedness: Boundedness,              // bounded / unbounded input?
}
```

None of this physical state exists in `LogicalPlan`.

## Why two representations are necessary

### 1) Many physical implementations per logical node

A single `LogicalPlan::Join` maps to multiple possible `ExecutionPlan` implementations depending on query structure, configuration, and statistics:

| Query characteristics | Physical operator |
|---|---|
| Equijoin, prefer hash | `HashJoinExec (Partitioned)` |
| Equijoin, one small side | `HashJoinExec (CollectLeft)` |
| Equijoin, inputs sorted | `SortMergeJoinExec` |
| Non-equi predicate | `NestedLoopJoinExec` |
| Single range inequality | `PiecewiseMergeJoinExec` |
| Both inputs unbounded | `SymmetricHashJoinExec` |
| No conditions | `CrossJoinExec` |

If `LogicalPlan` encoded the physical algorithm, every optimization and analysis rule would need to handle all six join variants. Since `LogicalPlan::Join` is algorithm-independent, logical rules only need to handle one node type.

### 2) Logical optimizer rules are simpler

A predicate pushdown rule operating on `LogicalPlan` handles:
- `LogicalPlan::Filter` + `LogicalPlan::Join` → push filter into join children

A physically-aware version would need to handle:
- `FilterExec` + `HashJoinExec` (Partitioned)
- `FilterExec` + `HashJoinExec` (CollectLeft)
- `FilterExec` + `SortMergeJoinExec`
- `FilterExec` + `NestedLoopJoinExec`
- ... etc.

The logical representation reduces the problem space dramatically.

### 3) Physical properties only become relevant during physical planning

`ExecutionPlan` carries `EquivalenceProperties` (known sort orders, functional equivalences from join conditions), `Partitioning` (how data is distributed), and `Boundedness`. These properties are:

- Irrelevant during logical optimization (predicate pushdown, CSE, projection pruning)
- Essential for physical optimization (`EnforceSorting`, `EnforceDistribution`, `JoinSelection`)

Keeping them in `ExecutionPlan` means logical rules never accidentally interact with physical properties. The `EnforceSorting` physical rule inserts `SortExec` nodes as needed; this is a concern of physical planning, not logical optimization.

### 4) The conversion is an explicit, extensible translation step

Source: [`datafusion/core/src/physical_planner.rs#L116`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/physical_planner.rs#L116)

```rust
pub trait PhysicalPlanner: Send + Sync {
    async fn create_physical_plan(
        &self,
        logical_plan: &LogicalPlan,
        session_state: &SessionState,
    ) -> Result<Arc<dyn ExecutionPlan>>;

    fn create_physical_expr(
        &self,
        expr: &Expr,
        input_dfschema: &DFSchema,
        session_state: &SessionState,
    ) -> Result<Arc<dyn PhysicalExpr>>;
}
```

This is a first-class extension API. Systems built on DataFusion can implement their own `PhysicalPlanner` to translate logical nodes into custom physical operators -- for example, Comet (Apache Spark accelerator) translates DataFusion logical plans into Spark-compatible execution plans. This is only possible because the boundary is an explicit trait with a clean signature.

The paper notes (Section 7.7): *"Users extend DataFusion by implementing the ExecutionPlan trait... DataFusion does not distinguish between user-defined and built-in ExecutionPlans during optimization and execution."*

## An example: from LogicalPlan::Filter to FilterExec

A SQL `WHERE status = 'active'` clause produces a `LogicalPlan::Filter`:

```rust
// LogicalPlan (abstract)
Filter {
    predicate: Expr::BinaryExpr {
        left: col("status"),
        op: Eq,
        right: lit("active"),
    },
    input: Arc<LogicalPlan::TableScan(...)>,
}
```

After physical planning, this becomes a `FilterExec`:

```rust
// ExecutionPlan (concrete)
FilterExec {
    predicate: Arc<dyn PhysicalExpr>,   // compiled, executable expression
    input: Arc<dyn ExecutionPlan>,      // concrete data source
    // inherits: partitioning (4 partitions), sort order, memory budget
}
```

And during `execute(partition=0, context)`, it returns a `SendableRecordBatchStream` that evaluates the physical predicate against each incoming `RecordBatch`. The `LogicalPlan::Filter` never knew it would run across 4 partitions in parallel.

---

## Summary

Two plan representations exist because they carry different information and serve different optimization phases. `LogicalPlan` is a compact, algorithm-independent representation of relational algebra that makes logical optimization simple and correct. `ExecutionPlan` is a concrete, executable operator with physical properties (partitioning, sort order, boundedness) that make physical optimization and execution possible. The separation reduces optimization complexity, enables multiple physical implementations per logical node, enables extension through a clean translation API, and follows the Volcano model's proven architectural principle.
