# Q8: Cardinality estimation is used to guide query optimization. Why is it so difficult to get accurate estimates, and how does DataFusion handle uncertainty in its statistics?

Cardinality estimation is hard because queries involve operations (filters, joins, aggregates) whose output sizes depend on data distribution, which is expensive to measure precisely and changes over time. DataFusion handles this fundamental uncertainty through an explicit `Precision<T>` type that tracks *confidence* alongside each estimate, enabling the optimizer to make decisions proportional to available evidence.

The paper acknowledges the importance of statistics for optimization:
> "Using a combination of the Catalog API and expression evaluation, Catalogs built with DataFusion use file metadata (such as minimum and maximum values) to avoid reading entire files or parts of files (e.g. Parquet Row Groups)."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 7.2

And in the context of join ordering:
> "Much of this difference is due to a suboptimal join order, and when we manually force a better join order, the performance of the two systems becomes similar."
>
> — Section 8.1 (on TPC-H performance gaps)

This direct admission -- that suboptimal cardinality estimates produce suboptimal join orders -- illustrates exactly why accurate cardinality estimation matters.

---

## Why cardinality estimation is fundamentally difficult

### 1) The independence assumption is almost always false

The simplest selectivity model assumes query predicates are independent:

`P(A=x AND B=y) = P(A=x) × P(B=y)`

In practice, columns are correlated. For example:
- `city = 'New York' AND state = 'NY'` -- these are nearly perfectly correlated
- `age >= 18 AND income > 30000` -- correlated in most real datasets

The independence assumption can cause estimates to be off by orders of magnitude for multi-column predicates.

### 2) Join fanout is hard to predict

For a join between tables R and S on key k:

`|R ⋈ S| ≈ |R| × |S| / max(distinct(R.k), distinct(S.k))`

This formula (the "distinct count heuristic") assumes uniform key distribution. In reality, a few values may appear far more frequently (Zipf distribution), making this estimate wildly optimistic or pessimistic.

Source: [`datafusion/physical-plan/src/joins/utils.rs#L464`](https://github.com/apache/datafusion/blob/main/datafusion/physical-plan/src/joins/utils.rs#L464)

```rust
/// Estimates cardinality for an inner join.
/// Uses the distinct count of join keys:
///   estimate = left_rows * right_rows / max(left_distinct_count, right_distinct_count)
///
/// This is a well-known heuristic, assuming uniform key distribution.
/// It is often inaccurate in practice when distributions are skewed.
pub fn estimate_join_cardinality(
    join_type: &JoinType,
    left_stats: &Statistics,
    right_stats: &Statistics,
    join_on: &JoinOn,
) -> Option<PartialJoinStatistics> {
    match join_type {
        JoinType::Inner | JoinType::Left | JoinType::Right | JoinType::Full => {
            let left_num_rows = left_stats.num_rows.get_value()?;
            let right_num_rows = right_stats.num_rows.get_value()?;

            // Find the minimum distinct count across all join key pairs
            let join_selectivity = join_on.iter().try_fold(1.0_f64, |acc, (l, r)| {
                let left_distinct = left_stats
                    .column_statistics
                    .get(l.index())?
                    .distinct_count
                    .get_value()?;
                let right_distinct = right_stats
                    .column_statistics
                    .get(r.index())?
                    .distinct_count
                    .get_value()?;
                let max_distinct = std::cmp::max(*left_distinct, *right_distinct) as f64;
                if max_distinct > 0.0 {
                    Some(acc / max_distinct)
                } else {
                    None
                }
            })?;

            let estimated_rows =
                ((left_num_rows * right_num_rows) as f64 * join_selectivity) as usize;
            Some(PartialJoinStatistics { num_rows: estimated_rows, .. })
        }
        _ => None,
    }
}
```

The code comment captures the fundamental problem: this heuristic *assumes uniform key distribution* and is *often inaccurate in practice when distributions are skewed*.

### 3) Statistics are expensive to collect and become stale

Collecting exact statistics (histograms, exact distinct counts) requires a full table scan. For large datasets:
- HyperLogLog for distinct counts introduces probabilistic error
- Histograms have limited bucket counts that can't represent bimodal distributions
- Statistics collected at table creation time become outdated as data is updated

### 4) Multi-predicate selectivity compounds errors

For a query with five predicates, even if each individual selectivity estimate is off by 2×, the compounded error can be 2^5 = 32×. Leis et al. (2015) showed that modern query optimizers routinely underestimate join cardinalities by orders of magnitude.

## DataFusion's solution: `Precision<T>` for tracked uncertainty

DataFusion uses an explicit `Precision<T>` wrapper instead of treating all statistics as equally reliable:

Source: [`datafusion/common/src/stats.rs#L31`](https://github.com/apache/datafusion/blob/main/datafusion/common/src/stats.rs#L31)

```rust
/// Represents a statistical value with its level of certainty.
#[derive(Clone, Debug, PartialEq, Eq, PartialOrd)]
pub enum Precision<T: Debug + Clone + PartialEq + Eq + PartialOrd> {  // L31
    /// The exact value is known (e.g., counted from all rows)
    Exact(T),
    /// An approximation is available (e.g., from metadata, heuristics)
    Inexact(T),
    /// No information is available at all
    Absent,
}
```

This three-level system enables the optimizer to distinguish between:
- **`Exact`**: statistics derived from a full scan, COUNT(*), or known schema facts
- **`Inexact`**: estimates from Parquet row group metadata, heuristics, or propagated estimates
- **`Absent`**: no useful information available

### The Statistics struct

Source: [`datafusion/common/src/stats.rs#L358`](https://github.com/apache/datafusion/blob/main/datafusion/common/src/stats.rs#L358)

```rust
pub struct Statistics {  // L358
    /// Number of rows in the output
    pub num_rows: Precision<usize>,
    /// Total byte size of the output
    pub total_byte_size: Precision<usize>,
    /// Per-column statistics
    pub column_statistics: Vec<ColumnStatistics>,
}

pub struct ColumnStatistics {
    /// Number of null values
    pub null_count: Precision<usize>,
    /// Maximum value (for range-based pruning)
    pub max_value: Precision<ScalarValue>,
    /// Minimum value (for range-based pruning)
    pub min_value: Precision<ScalarValue>,
    /// Number of distinct values (for join cardinality heuristic)
    pub distinct_count: Precision<usize>,
}
```

## How uncertainty propagates through operators

### Filter selectivity

Source: [`datafusion/physical-plan/src/filter.rs#L313`](https://github.com/apache/datafusion/blob/main/datafusion/physical-plan/src/filter.rs#L313)

```rust
fn statistics_helper(  // L313
    &self,
    predicate: &Arc<dyn PhysicalExpr>,
) -> Result<Statistics> {
    let input_stats = self.input.statistics()?;
    let selectivity = self.default_selectivity as f64 / 100.0;

    let num_rows = match input_stats.num_rows {
        Precision::Exact(n) => Precision::Inexact((n as f64 * selectivity) as usize),
        Precision::Inexact(n) => Precision::Inexact((n as f64 * selectivity) as usize),
        Precision::Absent => Precision::Absent,
    };
    // ...
}
```

Key observations:
1. Even if input `num_rows` is `Exact`, after applying a selectivity estimate the result becomes `Inexact` -- because the selectivity itself is an approximation
2. The `Absent` state propagates -- if we don't know input size, we can't estimate output size
3. The default selectivity fallback (e.g., 20%) is used when no better estimate exists

### Aggregate statistics

Source: [`datafusion/physical-plan/src/aggregates/mod.rs#L1041`](https://github.com/apache/datafusion/blob/main/datafusion/physical-plan/src/aggregates/mod.rs#L1041)

```rust
fn statistics(&self) -> Result<Statistics> {
    // For GROUP BY, we can only estimate output rows using group key distinct counts
    let group_count = self.group_by.expr().len();
    if group_count == 0 {
        // No GROUP BY: single output row
        return Ok(Statistics {
            num_rows: Precision::Exact(1),
            ..Statistics::new_unknown()
        });
    }

    // With GROUP BY: estimate from distinct counts of group key columns
    // but this is approximate; mark as Inexact
    let estimated_output = estimate_output_cardinality(&input_stats, group_count);
    Ok(Statistics {
        num_rows: Precision::Inexact(estimated_output),
        ..Statistics::new_unknown()
    })
}
```

A `GROUP BY` with no aggregation over all rows returns exactly 1 row -- this is `Exact`. But estimating how many groups a `GROUP BY user_id` produces requires knowing `DISTINCT(user_id)`, which is usually `Inexact`.

### Precision arithmetic

The `Precision<T>` type includes arithmetic that preserves the worst-case precision level:

```rust
impl Precision<usize> {
    pub fn add(self, other: Precision<usize>) -> Precision<usize> {
        match (self, other) {
            (Precision::Exact(a), Precision::Exact(b)) => Precision::Exact(a + b),
            (Precision::Exact(a), Precision::Inexact(b))
            | (Precision::Inexact(a), Precision::Exact(b))
            | (Precision::Inexact(a), Precision::Inexact(b)) => {
                Precision::Inexact(a + b)
            }
            _ => Precision::Absent,
        }
    }
}
```

`Exact + Exact = Exact`, but `Exact + Inexact = Inexact`. Uncertainty is contagious.

## How estimates drive optimizer decisions

### Join side selection

Source: [`datafusion/physical-optimizer/src/join_selection.rs#L70`](https://github.com/apache/datafusion/blob/main/datafusion/physical-optimizer/src/join_selection.rs#L70)

```rust
pub(crate) fn should_swap_join_order(
    left: &dyn ExecutionPlan,
    right: &dyn ExecutionPlan,
    config: &ConfigOptions,
) -> Result<bool> {
    let left_stats = left.partition_statistics(None)?;
    let right_stats = right.partition_statistics(None)?;
    // Prefer byte size if available; fall back to row count
    match (
        left_stats.total_byte_size.get_value(),   // get_value() returns None for Absent
        right_stats.total_byte_size.get_value(),
    ) {
        (Some(l), Some(r)) => Ok(l > r),
        _ => match (
            left_stats.num_rows.get_value(),
            right_stats.num_rows.get_value(),
        ) {
            (Some(l), Some(r)) => Ok(l > r),
            _ => Ok(false),  // no info: don't swap (conservative default)
        },
    }
}
```

`get_value()` returns `None` for `Precision::Absent`, so when statistics are missing the optimizer conservatively declines to reorder. This avoids making decisions based on no information.

### `AggregateStatistics` physical optimizer rule

When exact row counts are known (e.g., from a `COUNT(*)` that can be answered from metadata alone), `AggregateStatistics` short-circuits query execution entirely -- the aggregate answer is injected as a literal without scanning the table.

## What DataFusion does not yet have

The paper acknowledges limits (Section 8.1): join ordering quality is a known weakness. DataFusion currently lacks:
- Multi-column histograms (for correlated predicate estimation)
- Sketch-based distinct count propagation through arbitrary expressions
- Feedback-based statistics update from previous query executions
- Cost model for hash join memory spillover

These are active areas of development, not fundamental architectural gaps.

---

## Summary

Cardinality estimation is hard because real data distributions are skewed, correlated, and change over time -- defeating simple independence assumptions and distinct count heuristics. DataFusion addresses this honestly with `Precision<T>`: a three-level type (`Exact`, `Inexact`, `Absent`) that tracks estimation confidence and propagates uncertainty through the plan. Optimizer decisions use `get_value()` which returns `None` for absent statistics, so missing information causes conservative defaults rather than incorrect decisions. The TPC-H performance gap in the paper directly demonstrates that imperfect cardinality estimates produce suboptimal join orders -- a problem DataFusion acknowledges and is actively working to improve.
