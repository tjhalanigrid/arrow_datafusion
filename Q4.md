# Q4: DataFusion's optimizer applies logical transformations (like predicate pushdown) before physical planning. Why apply logical optimization before physical planning rather than interleaving them?

DataFusion uses a strict phase-separated optimization architecture: logical optimization completes fully before physical planning begins. This is a deliberate design choice with concrete benefits rooted in both architectural clarity and decades of database research.

The paper describes the architectural flow explicitly:
> "(1) Catalog and Data Sources provide schema and data layout and location information. (2) A Front End creates a tree of relational operators and expressions, called a LogicalPlan. (3) Optimizers rewrite the LogicalPlan and expressions to a more optimal form. (4) The LogicalPlan is lowered to an ExecutionPlan that includes characteristics of the intermediate results such as sort order, and specific algorithm selections. (5) Additional optimizers rewrite the ExecutionPlan to match available physical resources and data layout. (6) The ExecutionPlan is executed by Streams that incrementally produce results."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 5.1

## How the pipeline is implemented

The separation is explicit in `SessionState::create_physical_plan`:

Source: [`datafusion/core/src/execution/session_state.rs#L725`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/execution/session_state.rs#L725)

```rust
pub async fn create_physical_plan(
    &self,
    logical_plan: &LogicalPlan,
) -> datafusion_common::Result<Arc<dyn ExecutionPlan>> {
    let logical_plan = self.optimize(logical_plan)?;
    self.query_planner
        .create_physical_plan(&logical_plan, self)
        .await
}
```

`optimize()` runs the complete logical pipeline first. Only after that completes does physical planning begin. There is no alternation between the two.

## Why logical optimization runs first

### 1) Logical plans are simpler -- smaller optimization surface area

`LogicalPlan` has ~15 node variants: `Projection`, `Filter`, `Join`, `Aggregate`, `Sort`, `TableScan`, etc. Each variant is implementation-independent.

`ExecutionPlan` has dozens of concrete implementations: `HashJoinExec`, `SortMergeJoinExec`, `NestedLoopJoinExec`, `FilterExec`, `HashAggregateExec`, `DataSourceExec`, etc., each with varying partitioning modes, sort orders, and distribution requirements.

Logical optimizer rules only need to understand the small, abstract logical algebra. A predicate pushdown rule handles `LogicalPlan::Filter` + `LogicalPlan::Join` -- two variants. If it operated at the physical level, it would need to handle `FilterExec` + `HashJoinExec`, `FilterExec` + `SortMergeJoinExec`, `FilterExec` + `NestedLoopJoinExec`, `FilterExec` + `CrossJoinExec`, etc.

### 2) Logical rewrites create opportunities for physical planning

Many logical rewrites change what physical operators are even needed:

- **`EliminateCrossJoin`**: converts cross joins to inner joins -- eliminates the need for `CrossJoinExec`
- **`PushDownFilter`**: moves filters closer to scans -- enables `DataSourceExec` to use predicate pushdown and skip Parquet row groups (Section 6.8 of paper)
- **`DecorrelatePredicateSubquery`**: converts subqueries into joins -- creates new join nodes that need physical algorithm selection
- **`EliminateOuterJoin`**: converts outer joins to inner joins where provably safe -- enables hash join strategies unavailable for outer joins

These must complete before physical planning, otherwise physical operators would be created for logical patterns that the optimizer would later eliminate.

### 3) Physical decisions require information not available during logical optimization

Physical planning needs:
- `target_partitions` -- to choose between hash and sort-merge join
- `prefer_hash_join` config -- algorithm preference
- Statistics (row counts, byte sizes) -- for `JoinSelection`
- Memory thresholds -- for `PartitionMode` selection

During logical optimization, these physical resource decisions are correctly deferred.

### 4) Separation follows the Volcano model

DataFusion's design follows the Volcano execution model (Graefe, 1990, cited as [33] in the SIGMOD paper): *"DataFusion uses a pull-based streaming execution model and distributes work across multiple cores using Volcano-style exchange operators (viz. RepartitionExec)"* (Section 5.5).

The Volcano model naturally separates logical "what" from physical "how". DataFusion's optimizer pipeline applies this principle at the architectural level.

### 5) The Cascades alternative is more complex

Systems like SQL Server use a Cascades-style optimizer (Graefe, 1995) that interleaves logical and physical optimization in a single memo-based search. This can theoretically find better plans by exploring more combinations, but:

- Implementation complexity is substantially higher
- Debugging requires understanding the memo structure, not a linear pipeline
- Optimization time grows super-exponentially without aggressive pruning

DataFusion's phase-separated approach trades some plan quality for understandability, extensibility, and lower implementation cost -- consistent with its goal of being a practical, embeddable engine rather than an academic research prototype.

## What the phase separation enables

The paper's architecture (Figure 2) shows extension APIs at every phase boundary. Users can add `OptimizerRule` implementations (logical phase) or `PhysicalOptimizerRule` implementations (physical phase) independently, without understanding the other phase:

From the paper (Section 7.6):
> "DataFusion users add domain-specific optimizations such as input reordering and macro expansions by implementing OptimizerRules and PhysicalOptimizerRules, which rewrite LogicalPlan and ExecutionPlan trees, respectively, with the same APIs as built-in rewrites."

If the two phases were interleaved, custom rules would need to handle both logical and physical representations simultaneously.

---

## Summary

Logical optimization before physical planning keeps each phase tractable, allows logical rewrites to create opportunities for physical planners, defers physical resource decisions until they are needed, and follows the Volcano model's proven architecture. The tradeoff is that some cross-phase optimization opportunities (as in Cascades) are missed, but DataFusion compensates with a dedicated physical optimization pass.
