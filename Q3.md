# Q3: DataFusion implements multiple join algorithms (nested loop, hash join, sort-merge join). Why not use a single best algorithm, and what query characteristics determine which join to use?

There is no universally "best" join algorithm. Each algorithm has different performance characteristics depending on input sizes, sort order, memory budget, key distribution, and whether inputs are bounded or streaming. DataFusion implements **six physical join operators** and selects among them based on query structure, configuration, and runtime statistics.

The official paper describes the join philosophy as:
> "DataFusion automatically identifies equality (equi-join) predicates, heuristically reorders joins based on statistics, pushes predicates through joins (subject to OUTER join restrictions), introduces transitive join predicates, and picks the optimal physical join algorithm."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 6.4

### Hash Join

Builds a hash table on the left (build) side and probes with the right side. Implemented using vectorized hashing and collision checking similar to the MonetDB algorithm (SIGMOD paper ref [36]).

File: [`datafusion/physical-plan/src/joins/hash_join/exec.rs#L718`](https://github.com/apache/datafusion/blob/main/datafusion/physical-plan/src/joins/hash_join/exec.rs#L718) — `pub struct HashJoinExec`

```rust
pub struct HashJoinExec {
    /// left (build) side which gets hashed
    pub left: Arc<dyn ExecutionPlan>,
    /// right (probe) side which are filtered by the hash table
    pub right: Arc<dyn ExecutionPlan>,
    /// Set of equijoin columns from the relations: `(left_col, right_col)`
    pub on: Vec<(PhysicalExprRef, PhysicalExprRef)>,
    /// Filters which are applied while finding matching rows
    pub filter: Option<JoinFilter>,
    /// How the join is performed (`OUTER`, `INNER`, etc)
    pub join_type: JoinType,
    /// Partitioning mode: CollectLeft | Partitioned | Auto
    pub mode: PartitionMode,
    // ... (fetch, schema, metrics, etc.)
}
```

The `PartitionMode` determines parallelism:
- **`CollectLeft`**: build side is loaded into one shared hash table
- **`Partitioned`**: both sides are repartitioned by join key hash first
- **`Auto`**: resolved by the `JoinSelection` physical optimizer using statistics

### Sort-Merge Join

Merges two sorted streams on join keys — no hash table built. More memory-efficient when inputs are already sorted.

File: [`datafusion/physical-plan/src/joins/sort_merge_join/exec.rs#L109`](https://github.com/apache/datafusion/blob/main/datafusion/physical-plan/src/joins/sort_merge_join/exec.rs#L109) — `pub struct SortMergeJoinExec`

```rust
pub struct SortMergeJoinExec {
    /// Left sorted joining execution plan
    pub left: Arc<dyn ExecutionPlan>,
    /// Right sorting joining execution plan
    pub right: Arc<dyn ExecutionPlan>,
    /// Set of common columns used to join on
    pub on: JoinOn,
    /// Filters which are applied while finding matching rows
    pub filter: Option<JoinFilter>,
    /// How the join is performed
    pub join_type: JoinType,
    // ... (schema, metrics, left_sort_exprs, right_sort_exprs, sort_options, etc.)
}
```

DataFusion tracks sort orders on intermediate results (paper Section 6.7), so sort-merge join can be effectively "free" when upstream operations already sort on the join key.

## How the physical planner selects the join algorithm

The selection logic starts at line 1513 of:

File: [`datafusion/core/src/physical_planner.rs#L1513`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/physical_planner.rs#L1513) — decision tree

```rust
let prefer_hash_join =
    session_state.config_options().optimizer.prefer_hash_join;

let join: Arc<dyn ExecutionPlan> = if join_on.is_empty() {
    if join_filter.is_none() && *join_type == JoinType::Inner {
        // No conditions at all → CrossJoin
        Arc::new(CrossJoinExec::new(physical_left, physical_right))
    } else if num_range_filters == 1
        && total_filters == 1
        && !matches!(join_type, JoinType::LeftSemi | JoinType::RightSemi | ...)
        && session_state.config_options().optimizer.enable_piecewise_merge_join
    {
        // Single range predicate + PWMJ enabled → PiecewiseMergeJoinExec
        // (falls back to NestedLoopJoinExec on failure)
    } else {
        // Non-equi, general filter → NestedLoopJoinExec
        Arc::new(NestedLoopJoinExec::try_new(...)?)
    }
} else if target_partitions > 1 && repartition_joins && !prefer_hash_join {
    // Multi-partition + prefer sort-merge → SortMergeJoinExec
    Arc::new(SortMergeJoinExec::try_new(...)?)
} else if target_partitions > 1 && repartition_joins && prefer_hash_join && !null_aware {
    // Multi-partition + prefer hash (default) → HashJoinExec (Auto mode)
    Arc::new(HashJoinExec::try_new(..., PartitionMode::Auto, ...)?)
} else {
    // Single-partition or null-aware → HashJoinExec (CollectLeft)
    Arc::new(HashJoinExec::try_new(..., PartitionMode::CollectLeft, ...)?)
};
```

**Key config flags** (in `SessionConfig`):
- `optimizer.prefer_hash_join` (default: `true`) — chooses hash vs sort-merge
- `repartition_joins` — enables multi-partition strategies
- `enable_piecewise_merge_join` — opt-in experimental PWMJ

## Post-planning: JoinSelection physical optimizer

After initial planning, `JoinSelection` refines the choice using statistics.

File: [`datafusion/physical-optimizer/src/join_selection.rs#L70`](https://github.com/apache/datafusion/blob/main/datafusion/physical-optimizer/src/join_selection.rs#L70) — `should_swap_join_order`

```rust
pub(crate) fn should_swap_join_order(
    left: &dyn ExecutionPlan,
    right: &dyn ExecutionPlan,
    config: &ConfigOptions,
) -> Result<bool> {
    if !config.optimizer.join_reordering {
        return Ok(false);
    }
    // Compare total_byte_size first; fall back to num_rows
    let left_stats = left.partition_statistics(None)?;
    let right_stats = right.partition_statistics(None)?;
    match (
        left_stats.total_byte_size.get_value(),
        right_stats.total_byte_size.get_value(),
    ) {
        (Some(l), Some(r)) => Ok(l > r),
        _ => match (
            left_stats.num_rows.get_value(),
            right_stats.num_rows.get_value(),
        ) {
            (Some(l), Some(r)) => Ok(l > r),
            _ => Ok(false),
        },
    }
}
```

`JoinSelection` also:
- **Resolves `PartitionMode::Auto`** → `CollectLeft` or `Partitioned` using `hash_join_single_partition_threshold`
- **Converts to `SymmetricHashJoinExec`** when both inputs are `Unbounded`
- **Swaps build/probe sides** for `CrossJoinExec` and `NestedLoopJoinExec` when right side is smaller

## Why no single "best" algorithm

| Query characteristics | Best algorithm |
|---|---|
| Equijoin, large both sides, memory available | `HashJoinExec` (Partitioned) |
| Equijoin, one small build side | `HashJoinExec` (CollectLeft) |
| Equijoin, both inputs pre-sorted | `SortMergeJoinExec` |
| Non-equi predicate only | `NestedLoopJoinExec` |
| Single range inequality | `PiecewiseMergeJoinExec` |
| Inner join, no conditions | `CrossJoinExec` |
| Both inputs unbounded/streaming | `SymmetricHashJoinExec` |

- **Hash join**: fast, but materializes the build side — incompatible with unbounded inputs
- **Sort-merge**: memory-efficient when data is already sorted — avoids hash table allocation
- **Nested loop**: O(n×m) but the only correct choice for arbitrary non-equi predicates
- **Symmetric hash**: handles streaming inputs that cannot be materialized at all

The paper directly acknowledges the consequence of getting this wrong: *"there are several queries where DataFusion is well over 2x slower [than DuckDB], such as Q11, Q17, Q18, and Q21. Much of this difference is due to a suboptimal join order"* — Section 8.1.
