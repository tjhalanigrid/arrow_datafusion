# Q1: DataFusion is written in Rust but exposes a Python API. Why use Rust for the core engine rather than Python, and what architectural tradeoffs does this create for the Python integration?

DataFusion is written in Rust but has a Python API maintained in a separate repository. The primary motivation is that the *engine* needs systems-language properties (performance, safety, zero-cost abstractions, no runtime), while the *API layer* benefits from Python's ergonomics.

The official DataFusion paper states explicitly:
> "Rust features a low-level, yet safe memory management approach and C-like performance. It incorporates an innovative memory ownership model that mitigates many of the worst memory and thread safety challenges prevalent in traditional C/C++ programming. Rust programs are easy to embed in other systems as they do not require a language run-time and have C ABI compatibility."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 2.3

---

## Why Rust for the core engine

### 1) Performance and predictable resource use

DataFusion's execution is built around a *columnar*, *streaming*, multi-threaded model. Implementing this efficiently in CPython is hard due to interpreter overhead and the Global Interpreter Lock (GIL). Rust provides C-like performance with zero-cost abstractions.

DataFusion's physical operators must be `Send + Sync`, which Rust enforces at compile time:

Source: [`datafusion/physical-plan/src/execution_plan.rs#L96`](https://github.com/apache/datafusion/blob/main/datafusion/physical-plan/src/execution_plan.rs#L96)

```rust
pub trait ExecutionPlan: Debug + DisplayAs + Send + Sync {
    fn name(&self) -> &str;
    fn properties(&self) -> &Arc<PlanProperties>;
    fn execute(
        &self,
        partition: usize,
        context: Arc<TaskContext>,
    ) -> Result<SendableRecordBatchStream>;
    // ...
}
```

The `Send + Sync` bounds guarantee that plan nodes can be safely shared across threads. In CPython, the GIL would serialize this parallelism entirely.

The stream contract is defined as:

Source: [`datafusion/execution/src/stream.rs#L26`](https://github.com/apache/datafusion/blob/main/datafusion/execution/src/stream.rs#L26)

```rust
pub trait RecordBatchStream: Stream<Item = Result<RecordBatch>> {
    fn schema(&self) -> SchemaRef;
}

pub type SendableRecordBatchStream = Pin<Box<dyn RecordBatchStream + Send>>;
```

Streams run on Tokio, Rust's async runtime. As the paper notes: *"its combination of an efficient, work-stealing scheduler, first-class compiler support for automatic continuation generation, and exceptional performance makes it a compelling choice for CPU-intensive applications"* (Section 5.5.3).

### 2) Memory safety with zero-copy Arrow-first execution

Rust's ownership model enforces memory safety at compile time -- no garbage collector pauses, no dangling pointers. DataFusion uses Apache Arrow's C data interface for zero-copy data exchange at language boundaries:

Source: [`datafusion/ffi/src/arrow_wrappers.rs#L24`](https://github.com/apache/datafusion/blob/main/datafusion/ffi/src/arrow_wrappers.rs#L24)

```rust
use arrow::ffi::{FFI_ArrowArray, FFI_ArrowSchema, from_ffi, to_ffi};
```

Source: [`datafusion/ffi/src/arrow_wrappers.rs#L74`](https://github.com/apache/datafusion/blob/main/datafusion/ffi/src/arrow_wrappers.rs#L74)

```rust
#[repr(C)]
#[derive(Debug, StableAbi)]
pub struct WrappedArray {
    #[sabi(unsafe_opaque_field)]
    pub array: FFI_ArrowArray,
    pub schema: WrappedSchema,
}
```

This struct is `repr(C)` -- it uses a stable ABI and wraps Arrow's FFI array type. Python (and other language bindings) can share Arrow arrays with the Rust engine without any row-by-row copying.

### 3) Reusability across many embeddings

The paper highlights DataFusion's "LLVM parallel" (Section 4.1): just as LLVM's modular design enabled many language frontends to share a common compiler backend, DataFusion provides a shared query engine backend for many systems (InfluxDB 3.0, GreptimeDB, Coralogix, Comet for Spark). A Rust engine can be embedded in CLIs, JVM bridges (via JNI + Arrow), WebAssembly, and other language bindings without a language runtime dependency.

### 4) Ecosystem leverage

Rust's Cargo package manager makes adding DataFusion to a project as simple as one line in a configuration file (Section 2.3 of the paper). The Arrow, Parquet, and async ecosystem in Rust is mature and high-quality, meaning DataFusion benefits without reimplementing foundational components.

## Architectural tradeoffs for the Python integration

Choosing Rust for the core shifts complexity to the boundary between Python and the engine.

### 1) FFI boundary design: types, lifetimes, and ownership

Python objects cannot directly hold references into Rust memory without explicit lifetime management. Bindings wrap `Arc<...>` reference-counted handles. Extensions implemented from Python cannot directly implement Rust traits; instead they need an FFI-safe interface.

Source: [`datafusion/ffi/src/table_provider.rs#L58`](https://github.com/apache/datafusion/blob/main/datafusion/ffi/src/table_provider.rs#L58)

```rust
/// Each of the exposed structs in this crate is provided with a variant prefixed
/// with `Foreign`. This variant is designed to be used by the consumer of the
/// foreign code. The `Foreign` structs should _never_ access the `private_data`
/// fields. Instead they should only access the data returned through the function
/// calls defined on the `FFI_` structs.
```

The `FFI_TableProvider` struct exposes `TableProvider` across the FFI boundary with explicit function pointers:

```rust 
#[repr(C)]
#[derive(Debug, StableAbi)]
pub struct FFI_TableProvider {
    schema: unsafe extern "C" fn(provider: &Self) -> WrappedSchema,
    scan: unsafe extern "C" fn(
        provider: &Self,
        session: FFI_SessionRef,
        projections: ROption<RVec<usize>>,
        filters_serialized: RVec<u8>,   // <-- predicates as serialized bytes
        limit: ROption<usize>,
    ) -> FfiFuture<FFIResult<FFI_ExecutionPlan>>,
    // ...
}
```

Note that `filters_serialized` is a byte buffer (serialized protobuf), not a Rust type -- this is the "translation cost" of the FFI boundary.

### 2) Versioning and ABI compatibility

Native bindings require platform-specific wheels and careful versioning. DataFusion includes an explicit version check:

Source: [`datafusion/ffi/src/lib.rs#L60`](https://github.com/apache/datafusion/blob/main/datafusion/ffi/src/lib.rs#L60)

```rust
/// Returns the major version of the FFI implementation. If the API evolves,
/// we use the major version to identify compatibility over the unsafe boundary.
pub extern "C" fn version() -> u64 {
    let version_str = env!("CARGO_PKG_VERSION");
    let version = semver::Version::parse(version_str).expect("Invalid version string");
    version.major
}
```

### 3) Result streaming across the boundary

Even query results require explicit FFI handling. The stream of `RecordBatch` values is represented as:

Source: [`datafusion/ffi/src/record_batch_stream.rs#L39`](https://github.com/apache/datafusion/blob/main/datafusion/ffi/src/record_batch_stream.rs#L39)

```rust
pub struct FFI_RecordBatchStream {
    pub poll_next: unsafe extern "C" fn(
        stream: &Self,
        cx: &mut FfiContext,
    ) -> FfiPoll<ROption<FFIResult<WrappedArray>>>,
    pub schema: unsafe extern "C" fn(stream: &Self) -> WrappedSchema,
    pub release: unsafe extern "C" fn(arg: &mut Self),
    pub private_data: *mut c_void,
}
```

Each `RecordBatch` is converted to `WrappedArray` (Arrow FFI representation) before crossing to Python. When Python calls `next()` on the stream, it eventually calls `poll_next` through this interface.

### 4) Async/runtime bridging

DataFusion's API is async-first:

Source: [`datafusion/core/src/execution/context/mod.rs#L609`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/execution/context/mod.rs#L609)

```rust
pub async fn sql(&self, sql: &str) -> Result<DataFrame> {
    self.sql_with_options(sql, SQLOptions::new()).await
}
```

Python bindings must either expose async Python APIs, or run a Rust Tokio runtime internally and bridge it with Python event loops (e.g., using `pyo3-asyncio` or blocking on the runtime). This is a non-trivial integration challenge.

---

## Summary

Rust makes DataFusion fast, safe, and broadly embeddable -- achieving competitive performance with DuckDB on real workloads (ClickBench, TPC-H, H2O-G as shown in the SIGMOD paper Section 8) while remaining open and extensible. The Python integration tradeoff is the "systems boundary tax": stable ABI design, native wheel packaging, type serialization at the FFI boundary, and async bridging. These costs are manageable and are what enable DataFusion to serve as a shared engine for dozens of real-world systems.
