# Q10: DataFusion supports both SQL and a DataFrame API. Are these two separate paths through the engine, or do they converge somewhere? What are the tradeoffs of each interface?

SQL and the DataFrame API are **two separate front-ends that converge on the same `LogicalPlan` representation**. After the front-end, there is only one path: the same optimizer rules, the same physical planner, and the same streaming execution engine. The two APIs are genuinely equivalent in expressive power and performance.

The paper describes this explicitly:
> "In addition to SQL, DataFusion also offers a DataFrame API, modeled after pandas, for building query plans in a procedural style. The DataFrame API generates the same underlying LogicalPlan representation as the SQL API, and is optimized and executed the same way."
>
> — Lamb et al., *SIGMOD-Companion '24*, Section 5.3.3

---

## How SQL and DataFrame both produce a LogicalPlan

### SQL path

Source: [`datafusion/core/src/execution/context/mod.rs#L640`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/execution/context/mod.rs#L640)

```rust
pub async fn sql(&self, sql: &str) -> Result<DataFrame> {
    self.sql_with_options(sql, SQLOptions::new()).await
}

pub async fn sql_with_options(
    &self,
    sql: &str,
    options: SQLOptions,
) -> Result<DataFrame> {
    let plan = self.state().create_logical_plan(sql).await?;
    options.verify_plan(&plan)?;
    Ok(DataFrame::new(self.state(), plan))  // wraps LogicalPlan in DataFrame
}
```

`create_logical_plan` calls the SQL parser, runs the analyzer (type coercion, schema validation), and produces a `LogicalPlan`. That plan is immediately wrapped in a `DataFrame`. The SQL path *becomes* a DataFrame as its final step.

### DataFrame path

Source: [`datafusion/core/src/dataframe/mod.rs#L231`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/dataframe/mod.rs#L231)

```rust
/// A DataFrame represents a logical set of rows with a schema.
/// DataFrames are lazy: they do not execute until collected.
pub struct DataFrame {  // L231
    session_state: SessionStateRef,
    plan: LogicalPlan,   // the accumulated plan tree
}
```

A `DataFrame` is just a thin wrapper over a `LogicalPlan`. Every operation appends a new node to the tree:

Source: [`datafusion/core/src/dataframe/mod.rs#L592`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/dataframe/mod.rs#L592)

```rust
pub fn filter(self, predicate: Expr) -> Result<DataFrame> {
    let plan = LogicalPlanBuilder::from(self.plan)
        .filter(predicate)?
        .build()?;
    Ok(DataFrame {
        session_state: self.session_state,
        plan,
    })
}
```

`filter()` builds a new `LogicalPlan::Filter` node on top of the existing plan. No computation happens -- this is **deferred/lazy execution**. The pattern is the same for `select()`, `aggregate()`, `join()`, `sort()`, and `limit()`.

### Both execute the same way

Source: [`datafusion/core/src/dataframe/mod.rs#L1461`](https://github.com/apache/datafusion/blob/main/datafusion/core/src/dataframe/mod.rs#L1461)

```rust
pub async fn collect(self) -> Result<Vec<RecordBatch>> {  // L1461
    // Execute the DataFrame's LogicalPlan -- identical to executing SQL
    let task_ctx = Arc::new(TaskContext::from(&self.session_state));
    let plan = self
        .session_state
        .create_physical_plan(&self.plan)  // optimize + plan
        .await?;
    collect(plan, task_ctx).await  // stream all batches
}
```

`DataFrame::collect()` and SQL execution both call `create_physical_plan()` on the underlying `LogicalPlan`. There is no separate "DataFrame execution engine" -- they share exactly the same optimizer and execution path.

## The LogicalPlanBuilder API: a third path

For advanced use cases (custom query languages, programmatic plan construction), the `LogicalPlanBuilder` provides a Rust builder-style API for constructing plans directly:

Source: [`datafusion/expr/src/logical_plan/builder.rs#L128`](https://github.com/apache/datafusion/blob/main/datafusion/expr/src/logical_plan/builder.rs#L128)

```rust
pub struct LogicalPlanBuilder {  // L128
    plan: Arc<LogicalPlan>,
    /// Optional schema for the outer query (for correlated subqueries)
    outer_query_schema: Option<Arc<DFSchema>>,
}

impl LogicalPlanBuilder {
    pub fn scan(
        table_name: impl Into<TableReference>,
        table_source: Arc<dyn TableSource>,
        projection: Option<Vec<usize>>,
    ) -> Result<Self> { ... }

    pub fn filter(self, expr: impl Into<Expr>) -> Result<Self> { ... }
    pub fn project(self, expr: impl IntoIterator<Item = impl Into<Expr>>) -> Result<Self> { ... }
    pub fn aggregate(self, group_expr: Vec<Expr>, aggr_expr: Vec<Expr>) -> Result<Self> { ... }
    pub fn join(self, right: LogicalPlan, join_type: JoinType, ...) -> Result<Self> { ... }
    pub fn build(self) -> Result<LogicalPlan> { ... }
}
```

This is the same API that the `DataFrame` methods use internally. From the paper (Section 5.3.3): *"For more advanced uses, such as custom query languages, the LogicalPlanBuilder API offers a Rust builder-style interface for constructing plans directly."*

Real-world examples of this: InfluxDB IOx uses `LogicalPlanBuilder` to convert InfluxQL (a time-series query language) into `LogicalPlan` nodes, which are then optimized and executed by the standard DataFusion pipeline.

## Tradeoffs: SQL vs DataFrame

### SQL front-end

**Strengths:**
- Standard syntax familiar to any analyst or data engineer
- Can express complex queries compactly (`WINDOW`, `CTE`, `GROUPING SETS`)
- Parsed by `sqlparser-rs`, which supports ANSI SQL + many dialect extensions
- Easy to embed in tools that generate SQL strings

**Limitations:**
- String-based: no compile-time type checking of column names or types
- Dynamic queries require string manipulation or parameterization
- SQL parsing adds a small overhead (parse + analyze) vs. direct plan construction
- Some operations that are natural in code are awkward in SQL (e.g., applying a function to many columns)

### DataFrame front-end

**Strengths:**
- Compositional: each transformation is an explicit function call with typed arguments
- Programmatic: can build query plans with loops, conditionals, and abstractions
- Familiar to Python/pandas users (`filter`, `groupby`, `join`, `select`)
- No string parsing required for programmatic queries

**Limitations:**
- Not as portable as SQL (though the DataFusion DataFrame API follows pandas conventions)
- Complex SQL features (`RECURSIVE CTE`, `LATERAL JOIN`, specific window frame semantics) may be harder to express
- Less intuitive for ad-hoc analytical queries that users think of in SQL terms

### Equivalence in practice

The paper confirms they are fully equivalent (Section 5.3.3): *"The DataFrame API generates the same underlying LogicalPlan representation as the SQL API, and is optimized and executed the same way."*

This means:
- Performance is identical for equivalent queries
- Both APIs support UDFs, UDAFs, window functions, joins, aggregates
- Both benefit from all optimizer rules (predicate pushdown, CSE, column pruning, etc.)
- Both support the same extension points (custom `TableProvider`, custom functions)

## Serialization: a third dimension of equivalence

DataFusion also supports Substrait, an open cross-language serialization format for relational algebra. Both SQL and DataFrame plans can be serialized to Substrait and deserialized back, enabling cross-system plan exchange:

From the paper (Section 5.4.1): *"Libraries to (de)serialize these structures from/to bytes suitable for network transport, both using Protocol Buffers as well as Substrait."*

This means a Python system can construct a DataFrame plan, serialize it to Substrait bytes, send it over the network, and deserialize + execute it on a remote DataFusion engine -- regardless of which front-end was used to create the plan.

---

## Summary

SQL and DataFrame are two syntax options for building the same `LogicalPlan`. After the front-end layer, DataFusion has exactly one optimizer, one physical planner, and one execution engine. SQL is best for ad-hoc queries, portability, and tools that generate SQL strings. DataFrame is best for programmatic composition, type-safe query construction in code, and pandas-style workflows. Both are semantically equivalent, have identical performance for equivalent plans, and share all extension points. The convergence point is `LogicalPlan` -- the common language of all DataFusion front-ends.
